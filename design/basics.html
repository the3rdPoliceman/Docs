Basic Web Design
Early Stages of Design
Decide the Personality

Site Personality is important - you need to know where it lies on the scale of fun to serious before you can make any design decisions. It will affect font, colors, border radiuses among other things.

Develop Systems and Scales

To limit choices and reduce design anxiety, you'll need systems or scales for everything. It's not that you need to plan everything up front and stick to it â€” you'll still be able to tweak the scales later, and adjust components to use different entries from the scales. But having a list of 8 blue levels or 6 font sizes to choose from, is a lot easier than having unlimited.

Create a system for as many design elements as possible - these are probably a minimum.

  • Font face - One or two well chosen fonts
  • Font size - See font page for more detail
  • Font weight - 2 is normally enough. A 400/500 and a 600/700
  • Font color - Generally, 3 grey shades. Other font colors can be covered by color scales
  • Line height - Will scale inversely with font size, and may need to be tweaked for special situations and all uppercase
  • Color - 8 to 10 grey shades, and 6 to 8 shades of each color should be defined
  • Margin - related to font size
  • Padding - related to font size
  • Widths
  • Heights
  • Box shadows - define 4 or 5 levels that you can use from buttons (small shadow) up to panels, up to modal dialogs (large shadow)
  • Border radius - A scale of border radiuses, which will depend on component size, should be defined so you have consistency between components. Size will depend on the playfulness of your interface
  • Border width
  • Opacity
Start with a Feature

Don't try to design everything in advance. Come up with a feature, implement it and see what you think. Iterate the feature and the design

Designing features has a number of stages

  • Start with low detail, sketching without color
  • Design only what you need
  • Deal in the definite, don't design for possible future changes

Logical Design of a Page or Feature
Identify the information and actions

Initially, ignore the graphic design

Enumerate the information your feature needs

List the actions the user will be able to take

Prioritize and group information

Recognize the importance of each piece of information, remembering that high level information is not necessarily more important.
For example, on a bank account page you may have a title "Account Overview". This may seem important as it's the title, but to the user the account balance will probably be much more important - they already know they're on the account page.

Decide what bits of information are closely related, so these can be grouped together

Classify Actions by Importance

A Primary action is an action that a user is almost always expected to choose - Like a save button on a settings page, or an Add Item button on a shopping list. With these you'll want them to be very obvious. Here is a good explanation.

Secondary actions are obvious possibilities e.g. a Cancel button beside Save, and should be styled similarly to primary actions, but in a more subtle way. Maybe a less saturated color, or outlined rather than filled

Tertiary actions are expected to be used infrequently. Normally these will have very subtle styling - perhaps resembling a link, or just text

Visual Design of a Page or Feature
The Basic Rules

Group what is related, put space between what is unrelated

Emphasize what is important, deemphasize what is not important

Don't rely on font size to create hierarchy - there aren't enough sizes available anyway, and everything all the same font different sizes looks a mess. Using weight, color, background will improve the look and give more options

Spacing & Scaling

Start laying things out with too much space and remove extra, rather than starting cramped and adding

Don't let things expand beyond reason - you don't have to fill the screen

Not all elements should be fluid - some things just have a correct size, and should not expand or contract

Relative sizing doesn't always work at different scales e.g. having title text 3 times bigger than body may work on a monitor, but may be too much difference on a phone

Scaling buttons requires attention to button borders and padding

Creating Depth

To visually recreate depth on a flat screen the one basic fact to think about is that light comes from above. Things sticking out of a surface are light on top and dark at the bottom, the reverse for inset things. This can be emulated with box shadows and borders. To mimic both direct and ambient light, box shadows can be given 2 components

If elements are interactive (buttons, drag n drop etc) think of having different box shadows for each state to emulate the change. For example

  • A pressed button state with less shadow so it appears pushed in
  • An exaggerated box shadow on a dragged element will make it appear to float

If you're looking for a flatter UI, you can still use color to create a more subtle impression of depth. Light colours come forward, dark look inset. Another even more obvious clue to depth is when elements overlap

Using Labels

Avoid labelling if possible. It's probably necessary on text input fields, but if data can be presented and understood without labels then do so. For example, on a social network you are unlikely to see "Name: Bob Pratchitt". You'll just see the name.

On the other hand, if presenting a long list of fields such as product specifications, you will almost certainly need labels

Styling Links

Generally, a global style for links is not useful. How they are styled will depend on how important they are in the context, and how many links there are around. For example, video links in Youtube are styled to be much more obvious than the channel links, because people go to the videos more often

Styling Dangerous Actions

The temptation with dangerous actions is to put them in bright red. This is normally a bad idea - they shouldn't draw the eye away from more frequently used actions unless they are the primary action in the context. A better solution may be a low key delete button with a warning popup that makes the danger clear - like the GitHub system for deleting a repository

Finishing Touches
  • Custom checkbox, radio buttons and list icons are low effort upgrades
  • Accent borders are good, and can be used on tabs, modal, panels etc.
  • Empty lists etc. can look weird - think RepCoach without any exercises. A custom empty state - using an icon, explantation, or exaggerated call to action can be useful
Background Images

Cool background images with text over are nice, but often your text gets partially into them. This is because the background image is too dynamic

To solve this, filter your image to lighten, darken, colorize or whateverize it, and allow the text to stand out better to Alternatively, if you really don't want to mess with your photo, try using text shadow
